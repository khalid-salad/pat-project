       2             :   Solving memory overlap integer programs and bounded Diophantine equations with
       3             :   positive coefficients.
       4             : 
       5             :   Asking whether two strided arrays `a` and `b` overlap is equivalent to
       6             :   asking whether there is a solution to the following problem::
       7             : 
       8             :       sum(stride_a[i] * x_a[i] for i in range(ndim_a))
       9             :       -
      10             :       sum(stride_b[i] * x_b[i] for i in range(ndim_b))
      11             :       ==
      12             :       base_b - base_a
      13             : 
      14             :       0 <= x_a[i] < shape_a[i]
      15             :       0 <= x_b[i] < shape_b[i]
      16             : 
      17             :   for some integer x_a, x_b.  Itemsize needs to be considered as an additional
      18             :   dimension with stride 1 and size itemsize.
      19             : 
      20             :   Negative strides can be changed to positive (and vice versa) by changing
      21             :   variables x[i] -> shape[i] - 1 - x[i], and zero strides can be dropped, so
      22             :   that the problem can be recast into a bounded Diophantine equation with
      23             :   positive coefficients::
      24             : 
      25             :      sum(a[i] * x[i] for i in range(n)) == b
      26             : 
      27             :      a[i] > 0
      28             : 
      29             :      0 <= x[i] <= ub[i]
      30             : 
      31             :   This problem is NP-hard --- runtime of algorithms grows exponentially with
      32             :   increasing ndim.
      33             : 
      34             : 
      35             :   *Algorithm description*
      36             : 
      37             :   A straightforward algorithm that excludes infeasible solutions using GCD-based
      38             :   pruning is outlined in Ref. [1]. It is implemented below. A number of other
      39             :   algorithms exist in the literature; however, this one seems to have
      40             :   performance satisfactory for the present purpose.
      41             : 
      42             :   The idea is that an equation::
      43             : 
      44             :       a_1 x_1 + a_2 x_2 + ... + a_n x_n = b
      45             :       0 <= x_i <= ub_i, i = 1...n
      46             : 
      47             :   implies::
      48             : 
      49             :       a_2' x_2' + a_3 x_3 + ... + a_n x_n = b
      50             : 
      51             :       0 <= x_i <= ub_i, i = 2...n
      52             : 
      53             :       0 <= x_1' <= c_1 ub_1 + c_2 ub_2
      54             : 
      55             :   with a_2' = gcd(a_1, a_2) and x_2' = c_1 x_1 + c_2 x_2 with c_1 = (a_1/a_1'),
      56             :   and c_2 = (a_2/a_1').  This procedure can be repeated to obtain::
      57             : 
      58             :       a_{n-1}' x_{n-1}' + a_n x_n = b
      59             : 
      60             :       0 <= x_{n-1}' <= ub_{n-1}'
      61             : 
      62             :       0 <= x_n <= ub_n
      63             : 
      64             :   Now, one can enumerate all candidate solutions for x_n.  For each, one can use
      65             :   the previous-level equation to enumerate potential solutions for x_{n-1}, with
      66             :   transformed right-hand side b -> b - a_n x_n.  And so forth, until after n-1
      67             :   nested for loops we either arrive at a candidate solution for x_1 (in which
      68             :   case we have found one solution to the problem), or find that the equations do
      69             :   not allow any solutions either for x_1 or one of the intermediate x_i (in
      70             :   which case we have proved there is no solution for the upper-level candidates
      71             :   chosen). If no solution is found for any candidate x_n, we have proved the
      72             :   problem is infeasible --- which for the memory overlap problem means there is
      73             :   no overlap.
      74             : 
      75             : 
      76             :   *Performance*
      77             : 
      78             :   Some common ndarray cases are easy for the algorithm:
      79             : 
      80             :   - Two arrays whose memory ranges do not overlap.
      81             : 
      82             :     These will be excluded by the bounds on x_n, with max_work=1. We also add
      83             :     this check as a fast path, to avoid computing GCDs needlessly, as this can
      84             :     take some time.
      85             : 
      86             :   - Arrays produced by continuous slicing of a continuous parent array (no
      87             :     internal overlap), e.g., a=x[:,0,:], b=x[:,1,:]. The strides taken together,
      88             :     mapped positive, and duplicates then satisfy gcd(stride[0], .., stride[j]) =
      89             :     stride[j] for some ordering.
      90             : 
      91             :     In this case, for each x[i] at most one candidate exists, given that the
      92             :     algorithm runs with strides sorted from largest to smallest. The problem can
      93             :     be written as::
      94             : 
      95             :        sum a_j x_j ?= b = sum a_j z_j
      96             : 
      97             :        a_j = n_{j+1} * n_{j+2} * ... * n_d,  a_d = 1
      98             :        0 <= x_j <= u_j <= 2*n_j - 2
      99             :        0 <= z_j <= n_j - 1
     100             : 
     101             :     b is the offset of the last element of the second array from the start of
     102             :     the first.  z_j are uniquely determined because of the gcd property. For
     103             :     each x_j, the bounds at first sight allow x_j=z_j and x_j=z_j+n_j. However,
     104             :     u_j <= n_j - 1 + z_j, so that at most one candidate is left.
     105             : 
     106             :   - Two arrays with stride-incommensurate starting points. For example,
     107             :     a=x[:,::2], b=x[:,1::2].
     108             : 
     109             :     The base address difference is incommensurate with all strides, so that
     110             :     there are no solution candidates to consider. For itemsize != 1, similar
     111             :     result is obtained for x_{n-1}.
     112             : 
     113             :   The above cases cover arrays produced by typical slicing of well-behaved
     114             :   parent arrays. More generally, more difficult cases can result::
     115             : 
     116             :       x = np.arange(4*20).reshape(4, 20).astype(np.int8)
     117             :       a = x[:,::7]
     118             :       b = x[:,3::3]
     119             : 
     120             :       <=>
     121             : 
     122             :       20*x1 + 7*x2 + 3*x3 = 78    (= 3 + 3*20 + 5*3)
     123             :       0 <= x1 <= 6, 0 <= x2 <= 2, 0 <= x3 <= 5
     124             : 
     125             :   Non-overlapping in this case relies on x.shape[1] <= lcm(7, 3) = 21.  However,
     126             :   elimination of x1 does not restrict candidate values for x3, so the algorithm
     127             :   ends up considering all values x3=0...5 separately.
     128             : 
     129             :   The upper bound for work done is prod(shape_a)*prod(shape_b), which scales
     130             :   faster than work done by binary ufuncs, after broadcasting,
     131             :   prod(shape_a). The bound may be loose, but it is possible to construct hard
     132             :   instances where ufunc is faster (adapted from [2,3])::
     133             : 
     134             :       from numpy.lib.stride_tricks import as_strided
     135             :       # Construct non-overlapping x1 and x2
     136             :       x = np.zeros([192163377], dtype=np.int8)
     137             :       x1 = as_strided(x, strides=(36674, 61119, 85569), shape=(1049, 1049, 1049))
     138             :       x2 = as_strided(x[64023025:], strides=(12223, 12224, 1), shape=(1049, 1049, 1))
     139             : 
     140             :   To avoid such worst cases, the amount of work done needs to be capped. If the
     141             :   overlap problem is related to ufuncs, one suitable cap choice is to scale
     142             :   max_work with the number of elements of the array. (Ref. [3] describes a more
     143             :   efficient algorithm for solving problems similar to the above --- however,
     144             :   also it must scale exponentially.)
     145             : 
     146             : 
     147             :   *Integer overflows*
     148             : 
     149             :   The algorithm is written in fixed-width integers, and can terminate with
     150             :   failure if integer overflow is detected (the implementation catches all
     151             :   cases). Potential failure modes:
     152             : 
     153             :   - Array extent sum(stride*(shape-1)) is too large (for int64).
     154             : 
     155             :   - Minimal solutions to a_i x_i + a_j x_j == b are too large,
     156             :     in some of the intermediate equations.
     157             : 
     158             :     We do this part of the computation in 128-bit integers.
     159             : 
     160             :   In general, overflows are expected only if array size is close to
     161             :   NPY_INT64_MAX, requiring ~exabyte size arrays, which is usually not possible.
