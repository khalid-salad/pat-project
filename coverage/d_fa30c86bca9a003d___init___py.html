3Typing (:mod:`numpy.typing`) 

4============================ 

5 

6.. warning:: 

7 

8 Some of the types in this module rely on features only present in 

9 the standard library in Python 3.8 and greater. If you want to use 

10 these types in earlier versions of Python, you should install the 

11 typing-extensions_ package. 

12 

13Large parts of the NumPy API have PEP-484-style type annotations. In 

14addition a number of type aliases are available to users, most prominently 

15the two below: 

16 

17- `ArrayLike`: objects that can be converted to arrays 

18- `DTypeLike`: objects that can be converted to dtypes 

19 

20.. _typing-extensions: https://pypi.org/project/typing-extensions/ 

21 

22Mypy plugin 

23----------- 

24 

25A mypy_ plugin is distributed in `numpy.typing` for managing a number of 

26platform-specific annotations. Its function can be split into to parts: 

27 

28* Assigning the (platform-dependent) precisions of certain `~numpy.number` subclasses, 

29 including the likes of `~numpy.int_`, `~numpy.intp` and `~numpy.longlong`. 

30 See the documentation on :ref:`scalar types <arrays.scalars.built-in>` for a 

31 comprehensive overview of the affected classes. without the plugin the precision 

32 of all relevant classes will be inferred as `~typing.Any`. 

33* Removing all extended-precision `~numpy.number` subclasses that are unavailable 

34 for the platform in question. Most notable this includes the likes of 

35 `~numpy.float128` and `~numpy.complex256`. Without the plugin *all* 

36 extended-precision types will, as far as mypy is concerned, be available 

37 to all platforms. 

38 

39To enable the plugin, one must add it to their mypy `configuration file`_: 

40 

41.. code-block:: ini 

42 

43 [mypy] 

44 plugins = numpy.typing.mypy_plugin 

45 

46.. _mypy: http://mypy-lang.org/ 

47.. _configuration file: https://mypy.readthedocs.io/en/stable/config_file.html 

48 

49Differences from the runtime NumPy API 

50-------------------------------------- 

51 

52NumPy is very flexible. Trying to describe the full range of 

53possibilities statically would result in types that are not very 

54helpful. For that reason, the typed NumPy API is often stricter than 

55the runtime NumPy API. This section describes some notable 

56differences. 

57 

58ArrayLike 

59~~~~~~~~~ 

60 

61The `ArrayLike` type tries to avoid creating object arrays. For 

62example, 

63 

64.. code-block:: python 

65 

66 >>> np.array(x**2 for x in range(10)) 

67 array(<generator object <genexpr> at ...>, dtype=object) 

68 

69is valid NumPy code which will create a 0-dimensional object 

70array. Type checkers will complain about the above example when using 

71the NumPy types however. If you really intended to do the above, then 

72you can either use a ``# type: ignore`` comment: 

73 

74.. code-block:: python 

75 

76 >>> np.array(x**2 for x in range(10)) # type: ignore 

77 

78or explicitly type the array like object as `~typing.Any`: 

79 

80.. code-block:: python 

81 

82 >>> from typing import Any 

83 >>> array_like: Any = (x**2 for x in range(10)) 

84 >>> np.array(array_like) 

85 array(<generator object <genexpr> at ...>, dtype=object) 

86 

87ndarray 

88~~~~~~~ 

89 

90It's possible to mutate the dtype of an array at runtime. For example, 

91the following code is valid: 

92 

93.. code-block:: python 

94 

95 >>> x = np.array([1, 2]) 

96 >>> x.dtype = np.bool_ 

97 

98This sort of mutation is not allowed by the types. Users who want to 

99write statically typed code should instead use the `numpy.ndarray.view` 

100method to create a view of the array with a different dtype. 

101 

102DTypeLike 

103~~~~~~~~~ 

104 

105The `DTypeLike` type tries to avoid creation of dtype objects using 

106dictionary of fields like below: 

107 

108.. code-block:: python 

109 

110 >>> x = np.dtype({"field1": (float, 1), "field2": (int, 3)}) 

111 

112Although this is valid NumPy code, the type checker will complain about it, 

113since its usage is discouraged. 

114Please see : :ref:`Data type objects <arrays.dtypes>` 

115 

116Number precision 

117~~~~~~~~~~~~~~~~ 

118 

119The precision of `numpy.number` subclasses is treated as a covariant generic 

120parameter (see :class:`~NBitBase`), simplifying the annotating of processes 

121involving precision-based casting. 

122 

123.. code-block:: python 

124 

125 >>> from typing import TypeVar 

126 >>> import numpy as np 

127 >>> import numpy.typing as npt 

128 

129 >>> T = TypeVar("T", bound=npt.NBitBase) 

130 >>> def func(a: "np.floating[T]", b: "np.floating[T]") -> "np.floating[T]": 

131 ... ... 

132 

133Consequently, the likes of `~numpy.float16`, `~numpy.float32` and 

134`~numpy.float64` are still sub-types of `~numpy.floating`, but, contrary to 

135runtime, they're not necessarily considered as sub-classes. 

136 

137Timedelta64 

138~~~~~~~~~~~ 

139 

140The `~numpy.timedelta64` class is not considered a subclass of `~numpy.signedinteger`, 

141the former only inheriting from `~numpy.generic` while static type checking. 

142 

1430D arrays 

144~~~~~~~~~ 

145 

146During runtime numpy aggressively casts any passed 0D arrays into their 

147corresponding `~numpy.generic` instance. Until the introduction of shape 

148typing (see :pep:`646`) it is unfortunately not possible to make the 

149necessary distinction between 0D and >0D arrays. While thus not strictly 

150correct, all operations are that can potentially perform a 0D-array -> scalar 

151cast are currently annotated as exclusively returning an `ndarray`. 

152 

153If it is known in advance that an operation _will_ perform a 

1540D-array -> scalar cast, then one can consider manually remedying the 

155situation with either `typing.cast` or a ``# type: ignore`` comment.
